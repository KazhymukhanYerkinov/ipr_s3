# Цель 1: Улучшение производительности и рендеринга Flutter-приложений

## Теория — Flutter Rendering Pipeline

### Три дерева Flutter

Flutter строит UI через **три параллельных дерева**:

```
Widget Tree          Element Tree          RenderObject Tree
(описание UI)     (жизненный цикл)     (layout + paint)

MyApp               MyAppElement           RenderView
 └─ Scaffold         └─ ScaffoldElement      └─ RenderFlex
     └─ Column           └─ ColumnElement        └─ RenderFlex
         ├─ Text              ├─ TextElement           ├─ RenderParagraph
         └─ Button            └─ ButtonElement         └─ RenderSemanticsAnnotation
```

**Widget** — неизменяемое описание (blueprint). Создаётся часто (каждый build).
**Element** — привязка к месту в дереве. Живёт между build-ами. Решает: обновить или пересоздать.
**RenderObject** — реальные вычисления (layout, paint, hit testing). Дорогой.

### Как происходит rebuild

```
setState() / BlocBuilder
       │
       ▼
Widget.build() → новый Widget tree
       │
       ▼
Element.updateChild()
       │
       ├── Widget.canUpdate(old, new)?
       │   ├── Да (same runtimeType + key) → update RenderObject
       │   └── Нет → unmount old Element + mount new
       │
       └── identical(oldWidget, newWidget)?  ← вот почему const важен
           └── Да → SKIP (ничего не делаем)
```

### Event Loop и UI thread

Flutter — **однопоточный** (main isolate). Event loop обрабатывает:

```
┌─────────────────────────────────────────┐
│            Main Isolate                  │
│                                          │
│  Event Queue:  [tap] [future] [timer]   │
│       │                                  │
│       ▼                                  │
│  Microtask Queue:  [then] [then]        │
│       │                                  │
│       ▼                                  │
│  Frame Callback:  build → layout → paint │
│       │                                  │
│       ▼                                  │
│  Rasterizer → GPU → Экран               │
└─────────────────────────────────────────┘
```

Если между frame callbacks (16.6ms при 60fps) выполняется тяжёлая операция — **frame drop (jank)**.

### 1. `ListView.builder` / `GridView.builder` — ленивые списки

**Файл:** `lib/features/files/presentation/screens/home_screen.dart`

Файлы рендерятся через `FileGrid`, который внутри использует `GridView.builder`:

```dart
// FileGrid внутри HomeScreen
FileGrid(
  files: files,
  onFileTap: (file) => context.pushRoute(FileViewerRoute(...)),
  onFileDelete: (file) => _confirmDelete(context, file),
)
```

#### Почему так

`GridView.builder` создаёт виджеты **только для видимых элементов**. Если у нас 1000 файлов,
в памяти будет ~15–20 виджетов (видимые + буфер), а не все 1000.

#### Альтернатива (плохая)

```dart
// ❌ GridView(children: [...]) — создаёт ВСЕ виджеты сразу
GridView(
  children: files.map((f) => FileCard(file: f)).toList(),
)
```

При 1000 файлов — огромный расход памяти, долгий первый frame.

#### Под капотом

Flutter использует `SliverChildBuilderDelegate`, который вызывает builder лениво.
Когда пользователь скроллит, `RenderSliverList` вычисляет видимые индексы и вызывает
builder только для них. Невидимые элементы `dispose`'ятся и удаляются из Element tree.

```
Viewport (видимая область)
  └─ SliverList
       └─ SliverChildBuilderDelegate
            │  builder(context, 0)  → FileCard  ← создан
            │  builder(context, 1)  → FileCard  ← создан
            │  builder(context, 2)  → FileCard  ← создан
            │  ...
            │  builder(context, 15) → FileCard  ← создан (буфер)
            │  index 16..999        → НЕ СОЗДАН (экономия памяти)
```

#### Ещё пример — FolderTreeWidget

```dart
// lib/features/folders/presentation/widgets/folder_tree_widget.dart
ListView.builder(
  itemCount: folders.length,
  padding: const EdgeInsets.symmetric(horizontal: 8),
  itemBuilder: (context, index) => FolderNode(
    folder: folders[index],
    depth: 0,
    ...
  ),
)
```

### 2. `const` constructors — каноничные экземпляры

**Файлы:** практически везде в проекте.

```dart
// Экраны
const HomeScreen({super.key});
const _HomeView();

// Freezed states
const factory AuthState.initial() = AuthInitial;
const factory AuthState.loading() = AuthLoading;
const factory AuthState.unauthenticated() = Unauthenticated;

// Виджеты
const SizedBox(height: 12)
const Icon(Icons.folder_off_outlined, size: 48)
```

#### Почему так

Когда `build()` вызывается, Flutter сравнивает старый и новый Widget tree.
Если виджет создан через `const`, Dart гарантирует **один объект в памяти** (canonical instance).

```dart
// identical == true → Flutter ПОЛНОСТЬЮ пропускает rebuild
identical(const SizedBox(height: 12), const SizedBox(height: 12)) // true
```

#### Под капотом

На уровне Element tree: `Element.updateChild()` вызывает `Widget.canUpdate()`,
но ПЕРЕД этим проверяет `identical`. Если identical — даже `canUpdate` не вызывается:

```dart
// Flutter framework (element.dart, упрощённо)
void updateChild(Element? child, Widget? newWidget) {
  if (identical(child?.widget, newWidget)) return; // ← early exit!
  // ... дальше canUpdate, rebuild, etc.
}
```

#### Пример экономии

```
Без const:
  build() → создаёт new SizedBox(height: 12) → allocate → compare → update Element

С const:
  build() → const SizedBox(height: 12) → identical == true → SKIP
```

При 60 FPS × 100 виджетов = 6000 проверок/секунду. `const` экономит значительно.

### 3. `BlocConsumer` / `BlocSelector` — минимизация rebuilds

**Файл:** `lib/features/files/presentation/screens/home_screen.dart`

```dart
BlocConsumer<FilesBloc, FilesState>(
  listener: (context, state) {
    if (state is FilesError) {
      context.showFloatingSnackBar(state.message);
    }
  },
  builder: (context, state) {
    return switch (state) {
      FilesLoading() => const LoadingStateView(),
      FilesImporting(:final fileName) => LoadingStateView(message: ...),
      FilesLoaded(:final files) => files.isEmpty
          ? EmptyState(theme: theme)
          : FileGrid(files: files, ...),
      FilesError() => ErrorStateView(...),
      _ => const SizedBox.shrink(),
    };
  },
)
```

#### Почему `BlocConsumer`, а не отдельные

Объединяет `BlocListener` (side-effects: SnackBar) и `BlocBuilder` (UI) в **один** виджет.
Без него пришлось бы вложить `BlocListener` в `BlocBuilder` — лишняя вложенность.

#### Альтернатива — `BlocSelector`

Если нужно слушать **только часть** state:

```dart
// Перерендерит ТОЛЬКО когда список файлов изменился
BlocSelector<FilesBloc, FilesState, List<SecureFileEntity>>(
  selector: (state) => state is FilesLoaded ? state.files : [],
  builder: (context, files) => FileGrid(files: files),
)
```

Даже если `searchQuery` поменялся — `FileGrid` не перестроится.

#### Как BlocBuilder решает перестраивать

```dart
// Внутри BlocBuilder (упрощённо)
@override
void didUpdateWidget() {
  // buildWhen по умолчанию: previousState != currentState
  if (widget.buildWhen?.call(previousState, currentState) ?? true) {
    _widget = widget.builder(context, currentState);
  }
}
```

Можно передать `buildWhen`:

```dart
BlocBuilder<FilesBloc, FilesState>(
  buildWhen: (prev, curr) => prev.runtimeType != curr.runtimeType,
  builder: (context, state) => ...,
)
```

### 4. Разбивка build-функций на мелкие виджеты

`HomeScreen` разбит на:
- `_HomeView` — приватный виджет с основной логикой
- `SearchBarWidget` — поле поиска с debounce
- `SortDropdown` — выбор стратегии сортировки
- `FileGrid` — сетка файлов
- `FileCard` / `GridCard` — карточка файла
- `EmptyState` — пустой список
- `FileThumbnail` — превью файла

#### Почему

Каждый виджет — отдельный `Element` в Element tree. Когда `FilesBloc` state меняется:

```
HomeScreen (не перестраивается — stateless обёртка)
  └── _HomeView
       ├── SearchBarWidget (не зависит от FilesBloc state)
       ├── SortDropdown (не зависит от FilesBloc state)
       └── BlocConsumer ← ТОЛЬКО этот поддерев перестраивается
            └── FileGrid
                 └── FileCard × N
```

Если бы всё было в одном `build()`, при любом изменении state перестраивался бы **весь экран**.

### 6. Dispose паттерны — предотвращение memory leaks

```dart
// FileSearchService — корректное завершение Isolate
void dispose() {
  _workerSendPort?.send('close');
  _subscription?.cancel();
  _mainReceivePort?.close();
  _isolate?.kill(priority: Isolate.immediate);
  _resultController.close();
  _isolate = null;
  _workerSendPort = null;
}

// EncryptionQueue — очистка ресурсов
void dispose() {
  clear();
  _progressController.close();
}
```

#### Что утекает без dispose

- `StreamController` без `close()` → stream listener остаётся в памяти
- `Isolate` без `kill()` → background isolate живёт вечно (отдельный heap!)
- `StreamSubscription` без `cancel()` → callback ссылается на контекст
