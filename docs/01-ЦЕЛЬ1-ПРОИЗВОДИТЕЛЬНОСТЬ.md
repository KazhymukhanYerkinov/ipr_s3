# Цель 1: Производительность и рендеринг

## Теория (что спросят)

### Rendering Pipeline Flutter

Flutter строит UI через **три дерева**:

```
Widget Tree  →  Element Tree  →  RenderObject Tree
(описание)      (менеджер)       (рисование)
```

1. **Widget** — это просто описание (blueprint). Виджеты immutable и пересоздаются при каждом build()
2. **Element** — "менеджер", который связывает Widget с RenderObject. Элементы **переиспользуются** (не создаются заново при rebuild)
3. **RenderObject** — реально отвечает за layout, paint, hit-testing

**Ключевой момент для ассесмента**: когда вызывается `setState()` или BLoC эмитит новый стейт:
- Widget **пересоздаётся** (новый объект)
- Element **сравнивает** новый и старый Widget (canUpdate?)
- Если тип и key совпадают — Element обновляет RenderObject
- Если нет — Element удаляется и создаётся новый

### Что такое Jank?

**Jank** — это когда кадр рендерится дольше 16мс (для 60 FPS). Пользователь видит "подтормаживание".

Причины:
- Тяжёлые вычисления на main thread (шифрование, парсинг)
- Слишком много rebuilds (виджет перестраивается без необходимости)
- Большие списки без lazy rendering

### Event Loop

Dart однопоточный. Есть **Event Loop** с двумя очередями:
1. **Microtask queue** (приоритетная) — Future.value, scheduleMicrotask
2. **Event queue** — Timer, I/O, UI events

Async операции (Future, async/await) не создают нового потока — они просто откладывают выполнение.

## Что реализовано в проекте

### 1. ListView.builder (ленивые списки)

**Файл**: `lib/features/files/presentation/screens/home_screen.dart`

**Проблема**: если 1000 файлов — нельзя рисовать все сразу.

**Решение**: `ListView.builder` создаёт виджеты **только для видимых** элементов.

```dart
ListView.builder(
  itemCount: files.length,
  itemBuilder: (context, index) {
    final file = files[index];
    return RepaintBoundary(
      child: FileCard(file: file),
    );
  },
)
```

**Как объяснить**: "Обычный ListView создаёт все дочерние виджеты сразу. ListView.builder — лениво, только те что видны на экране + немного буфера. Это O(visible) вместо O(total)."

### 2. RepaintBoundary

**Файл**: `lib/features/files/presentation/widgets/file_list_view.dart`

**Проблема**: когда один виджет меняется — перерисовывается всё дерево.

**Решение**: `RepaintBoundary` создаёт **отдельный слой** (Layer). Flutter перерисовывает только изменившийся слой.

```dart
RepaintBoundary(
  child: FileCard(file: file),
)
```

**Как объяснить**: "RepaintBoundary говорит Flutter: 'этот поддеревья рисуй на отдельном слое'. Если карточка не изменилась — слой кэшируется и не перерисовывается. Это особенно полезно для карточек с превью изображений."

### 3. const Constructors

**Везде в проекте**

**Проблема**: если виджет не const, Flutter каждый раз создаёт новый объект.

**Решение**: `const` позволяет Flutter переиспользовать объект (один и тот же экземпляр):

```dart
// Плохо — новый объект каждый build()
return SizedBox(height: 16);

// Хорошо — один объект, переиспользуется
return const SizedBox(height: 16);
```

**Как объяснить**: "const виджеты не пересоздаются при rebuild. Flutter видит что это тот же объект и не трогает Element/RenderObject. Меньше работы для GC."

### 4. BlocSelector (минимизация rebuilds)

**Файл**: `lib/features/files/presentation/screens/home_screen.dart`

**Проблема**: `BlocBuilder` перестраивает виджет при **любом** изменении стейта.

**Решение**: `BlocSelector` подписывается только на часть стейта:

```dart
BlocSelector<FilesBloc, FilesState, ViewMode>(
  selector: (state) => state is FilesLoaded ? state.viewMode : ViewMode.list,
  builder: (context, viewMode) {
    // Перестроится ТОЛЬКО когда viewMode изменится
    return Icon(viewMode == ViewMode.list ? Icons.grid_view : Icons.list);
  },
)
```

**Как объяснить**: "BlocSelector слушает стейт, но перестраивает виджет только если возвращаемое значение изменилось. Например, если добавили файл — viewMode не изменился, кнопка не перестроится."

### 5. Вынос тяжёлых операций из main thread

**Подробнее в Цели 3**, но суть:

// ПЛОХО — блокирует UI на время шифрования

### 6. Dispose паттерны

**Файлы**: BLoC классы, FileSearchService

Всегда закрываем ресурсы:
- `StreamController` → `.close()`
- `Isolate` → `.kill()` + `ReceivePort.close()`
- `StreamSubscription` → `.cancel()`
- `TextEditingController` → `.dispose()`

```dart
// FileSearchService
void dispose() {
  _workerSendPort?.send('close');
  _subscription?.cancel();
  _mainReceivePort?.close();
  _isolate?.kill(priority: Isolate.immediate);
  _resultController.close();
}
```

## Возможные вопросы на ассесменте

**В**: Что такое RepaintBoundary и когда его использовать?
**О**: Создаёт отдельный layer для поддерева. Используем когда виджет сложный (изображение, анимация) и редко меняется, но его "соседи" часто перерисовываются. В проекте — на карточках файлов с превью.

**В**: Разница между ListView и ListView.builder?
**О**: ListView создаёт все children сразу (даже невидимые). ListView.builder — лениво, по мере скролла. Для больших списков builder экономит память и CPU.

**В**: Как Flutter определяет что виджет нужно перестроить?
**О**: Через Element.canUpdate() — сравнивает runtimeType и key старого и нового Widget. Если совпадают — Element обновляется. Если нет — старый Element удаляется, создаётся новый.

**В**: Что такое jank?
**О**: Когда кадр рендерится дольше 16мс (60 FPS) или 8мс (120 FPS). Пользователь видит "заикание" UI. Причины: тяжёлые операции на main thread, лишние rebuilds.

**В**: Зачем dispose()?
**О**: Чтобы избежать memory leaks. Не закрытый StreamController или Subscription продолжает держать ссылки на объекты, GC не может их собрать.
