# Вопросы и ответы для ассессмента

## Общая архитектура

### Q: Почему Clean Architecture, а не MVC/MVVM?

**A:** Clean Architecture разделяет код на 3 слоя (data → domain → presentation) с
**однонаправленными зависимостями**. Domain-слой не зависит ни от Flutter, ни от Hive,
ни от Firebase — чистый Dart. Это даёт:

- **Тестируемость:** domain-логику можно тестировать без Flutter framework
- **Заменяемость:** поменять Hive на SQLite → меняем только data-слой
- **Масштабируемость:** новый feature → копируем структуру папок

MVC/MVVM не запрещают зависимость бизнес-логики от UI-фреймворка. В Clean Architecture
это архитектурно невозможно (domain не импортирует flutter).

### Q: Зачем behaviors вместо одного Repository?

**A:** Interface Segregation Principle (I в SOLID).

Один `AuthRepository` с 7 методами → каждый Use Case зависит от ВСЕХ 7 методов,
даже если использует только один:

```dart
// ❌ Толстый интерфейс
class SignInUseCase {
  final AuthRepository repo; // зависит от signIn, signOut, verifyPin, setPin...
}
```

7 отдельных behaviors → каждый Use Case зависит только от своего:

```dart
// ✅ Тонкий интерфейс
class SignInUseCase {
  final SignInWithGoogleBehavior _behavior; // только signInWithGoogle()
}
```

При тестировании мокаем только нужный behavior, а не весь repository.

### Q: Почему Result\<T\>, а не dartz Either?

**A:** Читаемость. `Either<Failure, T>` из dartz — функциональный стиль Haskell/Scala.
В Dart выглядит чуждо:

```dart
// dartz
result.fold(
  (failure) => emit(AuthState.error(message: failure.message)),
  (user) => emit(AuthState.authenticated(user: user)),
);

// Наш Result
result.when(
  success: (user) => emit(AuthState.authenticated(user: user)),
  error: (failure) => emit(AuthState.error(message: failure.message)),
);
```

Плюс `result.isSuccess`, `result.value`, `result.failure` — простой property access.

### Q: Зачем runGuarded?

**A:** Убирает дублирование try/catch. Без него каждый метод AuthService:

```dart
// ❌ Без runGuarded — повторяется 7 раз
Future<Result<UserEntity>> signInWithGoogle() async {
  try {
    final user = await _remoteSource.signInWithGoogle();
    _logger.info('Signed in');
    return SuccessResult(user);
  } catch (e, stackTrace) {
    _logger.error('Sign-in failed', e, stackTrace);
    return ErrorResult(AuthFailure(message: 'Failed'));
  }
}

// ✅ С runGuarded — компактно
Future<Result<UserEntity>> signInWithGoogle() => runGuarded(
  action: () async {
    final user = await _remoteSource.signInWithGoogle();
    _logger.info('Signed in');
    return user;
  },
  onError: () => const AuthFailure(message: 'Failed'),
  logger: _logger,
  errorMessage: 'Sign-in failed',
);
```

## Производительность (Цель 1)

### Q: Что такое jank и как его обнаружить?

**A:** Jank — пропуск кадров. При 60fps каждый кадр = 16.6ms. Если build+layout+paint
занимает >16.6ms — frame drop. Пользователь видит «подёргивание».

Обнаружение:
1. **Flutter DevTools → Performance** — flame chart показывает длительность каждого frame
2. **Performance Overlay** (`showPerformanceOverlay: true`) — зелёные/красные bars
3. **Timeline Events** — какой именно виджет долго строится

### Q: Как RepaintBoundary помогает?

**A:** Создаёт изолированный слой рисования. Без него при изменении одного FileCard
перерисовывается весь parent (Column/ListView). С RepaintBoundary — только этот FileCard.

```dart
RepaintBoundary(
  child: FileCard(file: file), // перерисовка только внутри этой boundary
)
```

Особенно эффективно для карточек с анимациями или тяжёлыми виджетами (изображения).

### Q: Когда НЕ нужен const?

**A:** Когда виджет принимает **runtime-данные**:

```dart
// ❌ Нельзя const — зависит от runtime
Text(file.name)            // file.name меняется
Icon(getIcon(file.type))   // зависит от типа файла

// ✅ Можно const — всегда одинаковый
const SizedBox(height: 12)
const Icon(Icons.folder)
```

## Isolate (Цель 3)

### Q: Можно ли обновлять UI из Isolate?

**A:** Нет. Isolate имеет свой heap, нет доступа к Widget tree main isolate.
Единственный способ — отправить данные через `SendPort`, main isolate обновит UI.

### Q: Чем compute отличается от async/await?

**A:** `async/await` — **не создаёт новый поток**. Это кооперативная многозадачность
в пределах одного event loop. Если функция CPU-bound (шифрование) — она блокирует
event loop, даже с `await`.

```dart
// ❌ async/await НЕ помогает для CPU-bound
Future<Uint8List> encrypt(Uint8List data) async {
  return aes.encrypt(data); // блокирует main thread 500ms!
}

// ✅ compute переносит в другой Isolate
Future<Uint8List> encrypt(Uint8List data) async {
  return compute(encryptBytes, params); // main thread свободен
}
```

`async/await` помогает только для I/O-bound задач (HTTP, file read) — они не
блокируют event loop, потому что ждут ОС.

### Q: Почему нельзя передать closure в Isolate?

**A:** Closure захватывает контекст (переменные из scope). Этот контекст живёт
в heap main isolate. Новый Isolate = новый heap = нет доступа к контексту.

```dart
class Service {
  final Key _key; // живёт в heap main isolate

  void doWork() {
    compute(_encrypt, data); // ❌ _encrypt ссылается на this._key
  }

  Uint8List _encrypt(Uint8List data) {
    return aes.encrypt(data, _key); // this._key недоступен в другом Isolate
  }
}
```

Решение: передать ВСЕ данные через параметр (EncryptionParams).

## Безопасность (Цель 4)

### Q: Чем Keychain отличается от Keystore?

**A:**
- **iOS Keychain** — системное хранилище, зашифрованное Secure Enclave. Данные
  привязаны к конкретному приложению (через entitlement). При удалении приложения
  данные остаются (можно настроить удаление).
- **Android Keystore** — аппаратный модуль безопасности (TEE). Ключи **никогда не покидают**
  TEE — шифрование/дешифрование выполняется внутри. Даже root не даёт ключ.

### Q: Что если пользователь сделает jailbreak/root?

**A:**
- **Keychain/Keystore** — по-прежнему защищены (hardware-backed)
- **Файлы на диске** — зашифрованы AES-256, без ключа = мусор
- **PIN hash** — в secure storage, не в SharedPreferences
- **Логи** — маскированы, в release только errors

Полной защиты от root нет (пользователь может подключить debugger к процессу),
но мы максимально усложняем атаку.

### Q: Почему OAuth 2.0, а не обычный email/password?

**A:**
1. **Мы не храним пароли** — Google хранит и защищает
2. **MFA бесплатно** — Google аккаунты поддерживают 2FA
3. **Нет вектора атаки** на наш сервер — у нас нет backend'а
4. **Проверенная инфраструктура** — Google Sign-In SDK, Firebase Auth

## Паттерны (Цель 5)

### Q: Как Strategy связан с Open-Closed Principle?

**A:** «Класс открыт для расширения, закрыт для модификации».

Добавить новую сортировку = создать новый класс `SortByTags implements SortStrategy`.
Не нужно менять `CommandManager`, `FilesBloc`, или другие стратегии.

### Q: Чем Command отличается от обычного метода?

**A:** Command — это **объект**, а не вызов метода. Объект можно:
1. **Сохранить** в стек (undo/redo history)
2. **Передать** как параметр
3. **Отложить** выполнение
4. **Сериализовать** (сохранить в файл для replay)

Обычный метод `deleteFile()` — вызвал и забыл, undo невозможно.

### Q: Зачем abstract class FilePreviewFactory, если всё в static методе?

**A:** `abstract class` запрещает создание экземпляров — это namespace для фабричного метода.
Можно было бы использовать обычный класс с `factory` constructor, но static метод проще
и нагляднее. В Flutter это распространённый паттерн для Simple Factory.

## Коллекции (Цель 6)

### Q: Когда List лучше LinkedList?

**A:** Почти всегда. `List` выигрывает по:
- Индексному доступу: O(1) vs O(n)
- Cache locality: элементы лежат рядом в памяти, CPU cache эффективнее
- Простоте API

`LinkedList` лучше **только** когда нужен O(1) `unlink` из середины/начала.
В нашем случае — undo стек с лимитом 50 и удалением первого элемента.

### Q: Зачем fold, если есть обычный for-цикл?

**A:** `fold` — **декларативный**: видно намерение «сверни в число». Цикл — **императивный**:
нужно читать тело. Для простых агрегаций (сумма, подсчёт) fold читаемее.
Для сложной логики с break/continue — цикл лучше.

### Q: Как hashCode работает в Set?

**A:** Set использует hash table. При `add(element)`:
1. Вычисляет `element.hashCode` → число
2. `number % buckets.length` → индекс bucket'а
3. В bucket'е ищет через `==` — если нашёл, не добавляет (дубликат)

Если `hashCode` плохой (всегда 0) — все элементы в одном bucket'е → O(n) поиск.
Хороший `hashCode` равномерно распределяет → O(1) amortized.

## Native (Цель 7)

### Q: Почему CRC32 через FFI, а не Dart-пакет?

**A:** Демонстрация FFI + производительность. C быстрее Dart в 5× на больших данных
(нет GC overhead, нативная оптимизация). Для 10 МБ файла: Dart ~440ms vs C ~80ms.

Для продакшена можно использовать `crc32c` пакет. Но мы показываем **механизм FFI**.

### Q: Что произойдёт если забыть malloc.free?

**A:** Memory leak. Dart GC не знает о нативной памяти. Каждый вызов crc32 без free
= +data.length bytes утечки. При частых вызовах (1000 файлов) — МБ утечки, OOM crash.

### Q: MethodChannel синхронный или асинхронный?

**A:** **Асинхронный** на Dart-стороне (`Future`). На нативной стороне — синхронный
(handler вызывается в main thread). Если нативный код долгий (>16ms),
нужно вынести в background thread (AsyncTask на Android, DispatchQueue на iOS).

Наши методы (батарея, storage) быстрые (~1ms), поэтому OK в main thread.

## CI/CD (Цель 2)

### Q: Почему build-android и build-ios параллельны?

**A:** Оба зависят от `quality`, но не друг от друга. GitHub Actions автоматически
запускает их параллельно. Вместо ~80 мин последовательно → ~40 мин параллельно.

### Q: Что такое GitHub Secrets и зачем?

**A:** Зашифрованные переменные окружения, доступные только в workflow.
Не видны в логах (маскируются `***`). Не доступны в fork PR (безопасность).

Используем для SONAR_TOKEN. В продакшене — для signing keys, Firebase tokens.

### Q: Зачем cancel-in-progress?

**A:** Если я запушил 3 коммита за минуту, запустится только **последний** pipeline.
Предыдущие два отменятся — экономим ресурсы и время. Промежуточные коммиты
не нужно билдить, важен только финальный.
