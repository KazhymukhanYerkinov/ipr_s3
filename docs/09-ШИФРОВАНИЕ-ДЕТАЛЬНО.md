## AES-256-CBC — алгоритм шифрования файлов

### Что такое AES

AES (Advanced Encryption Standard) — **симметричный** блочный шифр.
Принят NIST в 2001 году, заменил DES. Используется везде: TLS, WPA2, BitLocker, FileVault.

- **Симметричный:** один ключ для шифрования и дешифрования
- **Блочный:** обрабатывает данные блоками по 16 байт (128 бит)
- **AES-256:** длина ключа 256 бит (32 байта)

### AES-256 vs AES-128 vs AES-192

| Вариант | Длина ключа | Раунды | Безопасность |
|---------|-------------|--------|--------------|
| AES-128 | 128 бит | 10 | 2^128 комбинаций |
| AES-192 | 192 бит | 12 | 2^192 комбинаций |
| AES-256 | 256 бит | 14 | 2^256 комбинаций |

2^256 ≈ 10^77 — больше чем атомов во Вселенной (~10^80).
Brute force невозможен даже с квантовыми компьютерами (Grover's algorithm снижает до 2^128).

### Режим CBC (Cipher Block Chaining)

```
Plaintext:  [Block 1] [Block 2] [Block 3] [Block 4]
                │          │          │          │
                ▼          ▼          ▼          ▼
           IV──⊕    ┌────⊕    ┌────⊕    ┌────⊕
                │   │     │   │     │   │     │
                ▼   │     ▼   │     ▼   │     ▼
           AES_Enc  │  AES_Enc│  AES_Enc│  AES_Enc
                │   │     │   │     │   │     │
                ▼   │     ▼   │     ▼   │     ▼
           [Cipher 1]──┘[Cipher 2]──┘[Cipher 3]──┘[Cipher 4]
```

**Chaining:** каждый зашифрованный блок XOR'ится со следующим plaintext блоком.
Одинаковые plaintext блоки → **разные** cipher блоки (из-за chain).

**IV (Initialization Vector):** случайные 16 байт для первого блока.
Каждый файл шифруется с **уникальным IV** → один файл дважды зашифрованный
даёт **разный** ciphertext.

### Почему CBC, а не ECB

**ECB (Electronic Codebook)** — каждый блок шифруется независимо:

```
ECB: одинаковые блоки → одинаковый ciphertext
  "AAAA" "AAAA" "BBBB" → "xxxx" "xxxx" "yyyy"
                                  ^^^^   ^^^^
                               видны паттерны!

CBC: каждый блок зависит от предыдущего:
  "AAAA" "AAAA" "BBBB" → "abcd" "efgh" "ijkl"
                               ^^^^   ^^^^
                            нет паттернов
```

ECB на изображениях сохраняет контуры (знаменитый пример с пингвином Linux).
CBC — каждый блок зависит от всех предыдущих → нет видимых паттернов.

### Альтернативы CBC

| Режим | Плюсы | Минусы |
|-------|-------|--------|
| **CBC** (наш выбор) | Простой, надёжный, широко поддерживается | Последовательный (не параллелится) |
| **CTR** (Counter) | Параллельный, быстрее на multicore | Уязвим к nonce reuse |
| **GCM** (Galois/Counter) | Параллельный + аутентификация (AEAD) | Сложнее, нужен unique nonce |
| **XTS** | Стандарт для disk encryption | Сложнее реализовать |

Для нашего проекта CBC — оптимальный баланс простоты и безопасности.
Для банковского продакшена — GCM (AEAD = encryption + integrity в одном).

## Реализация в коде

### Шифрование

**Файл:** `lib/features/files/data/services/file_encryption_service.dart`

```dart
Uint8List encryptBytes(EncryptionParams params) {
  final key = enc.Key(params.key);                           // 32 bytes key
  final iv = enc.IV.fromSecureRandom(16);                    // Random 16 bytes IV
  final encrypter = enc.Encrypter(enc.AES(key, mode: enc.AESMode.cbc));

  final encrypted = encrypter.encryptBytes(params.data, iv: iv);

  // Формат: [IV (16 bytes)] + [Encrypted Data]
  final result = Uint8List(iv.bytes.length + encrypted.bytes.length);
  result.setRange(0, iv.bytes.length, iv.bytes);
  result.setRange(iv.bytes.length, result.length, encrypted.bytes);
  return result;
}
```

#### Почему IV хранится вместе с данными

IV — **не секрет**. Его задача — обеспечить уникальность шифрования (один файл дважды
зашифрованный → разный ciphertext). Для расшифровки IV нужен, поэтому сохраняем рядом.

Формат на диске:

```
[IV: 16 bytes][Encrypted data: N bytes]
│             │
└─ Не секрет  └─ Без ключа = мусор
```

### Дешифрование

```dart
Uint8List decryptBytes(EncryptionParams params) {
  final key = enc.Key(params.key);
  final ivBytes = params.data.sublist(0, 16);           // Первые 16 bytes = IV
  final encryptedData = params.data.sublist(16);         // Остальное = encrypted

  final iv = enc.IV(ivBytes);
  final encrypter = enc.Encrypter(enc.AES(key, mode: enc.AESMode.cbc));

  final decrypted = encrypter.decryptBytes(
    enc.Encrypted(encryptedData),
    iv: iv,
  );
  return Uint8List.fromList(decrypted);
}
```

### Генерация ключа

```dart
// Hive.generateSecureKey() — 32 random bytes через SecureRandom
final newKey = Hive.generateSecureKey(); // List<int>, length = 32
```

`SecureRandom` использует:
- **Android:** `/dev/urandom` (Linux CSPRNG)
- **iOS:** `SecRandomCopyBytes` (Security framework)

Оба — криптографически безопасные PRNG (не `Random()` из dart:math!).

## Целостность файлов — CRC32

### Зачем нужен checksum

Зашифрованный файл на диске может повредиться:
- Bad sector на flash-памяти
- Неполная запись (crash во время записи)
- Повреждение файловой системы

CRC32 обнаруживает повреждения:

```dart
// При импорте:
final encryptedBytes = await _encryptionService.encrypt(bytes);
final checksum = _nativeHashService.crc32(encryptedBytes);
// Сохраняем checksum в metadata

// При расшифровке:
final encryptedBytes = await _localSource.readEncryptedFile(path);
final currentChecksum = _nativeHashService.crc32(encryptedBytes);
if (currentChecksum != entity.checksum) {
  // Файл повреждён!
  return ErrorResult(const FileFailure(
    message: 'File integrity check failed — data may be corrupted',
  ));
}
```

### CRC32 vs SHA-256 для integrity

| Алгоритм | Скорость | Защита от подделки | Использование |
|----------|---------|-------------------|---------------|
| **CRC32** | Очень быстрая | ❌ Нет (легко подделать) | Детектирование **случайных** повреждений |
| **SHA-256** | Медленнее в 10× | ✅ Да (криптографический) | Защита от **целенаправленной** подделки |

Мы используем CRC32 потому что:
1. Защита от подделки не нужна (файлы локальные, злоумышленник уже внутри приложения)
2. CRC32 через C FFI — очень быстрый (~8ms для 10 МБ)
3. Задача — обнаружить **случайные** повреждения файловой системы

## Поток шифрования файла (end-to-end)

```
Пользователь выбирает файл
       │
       ▼
FilePicker → bytes (Uint8List)
       │
       ▼
FileTypeResolver → определяет тип (image/pdf/text)
       │
       ▼
┌──────┴──────┐
│  Если image  │
│  → compute() │
│  thumbnail   │
│  → encrypt   │
│  thumbnail   │
└──────┬──────┘
       │
       ▼
compute(encryptBytes, params)  ← Isolate (не блокирует UI)
  │
  │  Внутри Isolate:
  │  1. Key = 32 bytes (переданы через params)
  │  2. IV = random 16 bytes
  │  3. AES-256-CBC encrypt
  │  4. Result = IV + encrypted
  │
  ▼
CRC32 checksum (FFI) ← C-код, быстрый
       │
       ▼
Сохранение на диск:
  secure_files/{fileId}.enc ← зашифрованные байты
  secure_files/thumbs/{fileId}.enc ← зашифрованный thumbnail
       │
       ▼
Сохранение metadata в Hive (encrypted box):
  SecureFileDto {
    id, name, type, size,
    encryptedPath, thumbnailPath,
    checksum, createdAt, updatedAt,
    tags, folderId
  }
       │
       ▼
FilesBloc → emit(FilesState.loaded)
```

```
Пользователь тапает на файл
       │
       ▼

       │
       ▼

       ▼
```

## Безопасность ключа — детали

### Жизненный цикл ключа

```
Первый запуск:
  Hive.generateSecureKey() → 32 random bytes
       │
       ▼
  base64Url.encode(key) → String
       │
       ▼
  flutter_secure_storage.write(key: "hive_encryption_key", value: encoded)
       │
       ▼
  Android: → Android Keystore → TEE/StrongBox → AES-encrypt → SharedPreferences
  iOS:     → Keychain → Secure Enclave → AES-GCM encrypt → Keychain DB

Последующие запуски:
  flutter_secure_storage.read(key: "hive_encryption_key")
       │
       ▼
  Android: SharedPreferences → AES-decrypt в TEE → plaintext key
  iOS:     Keychain DB → AES-GCM decrypt в Secure Enclave → plaintext key
       │
       ▼
  base64Url.decode(encoded) → Uint8List (32 bytes)
       │
       ▼
  Используем для Hive + файлового шифрования
```

### Что делает flutter_secure_storage под капотом

**Android:**
1. Генерирует master key в Android Keystore (TEE)
2. Шифрует наш encryption key мастер-ключом
3. Сохраняет зашифрованный ключ в EncryptedSharedPreferences
4. При чтении: расшифровывает в TEE, возвращает plaintext

**iOS:**
1. Сохраняет в Keychain с `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`
2. Keychain зашифрован Secure Enclave (AES-256-GCM)
3. Данные доступны только пока устройство разблокировано
4. Привязаны к конкретному устройству (не синхронизируются)

## EncryptionQueue — последовательная обработка

### Зачем очередь

Если пользователь импортирует 10 файлов одновременно:
- **Без очереди:** 10 параллельных `compute()` → 10 Isolate → OOM
- **С очередью:** 1 файл за раз, остальные ждут в Queue → предсказуемый memory usage

### Реализация

```dart
final Queue<FileTask> _queue = Queue<FileTask>();

void enqueue(FileTask task) {
  _queue.addLast(task);
  if (!_isProcessing) _processQueue();
}

Future<void> _processQueue() async {
  _isProcessing = true;
  while (_queue.isNotEmpty) {
    final task = _queue.removeFirst();  // FIFO
    try {
      final encrypted = await _encryptionService.encrypt(task.data);
      _progressController.add(QueueProgress(
        completedCount: _completedCount,
        totalCount: total,
        isComplete: _queue.isEmpty,
        encryptedData: encrypted,
      ));
    } catch (e) {
      // Ошибка одного файла не останавливает очередь
      _progressController.add(QueueProgress(..., encryptedData: null));
    }
  }
  _isProcessing = false;
}
```

### Отказоустойчивость

Если шифрование одного файла упало → `catch` ловит → progress без `encryptedData` →
следующий файл из очереди обрабатывается. Очередь не останавливается.
