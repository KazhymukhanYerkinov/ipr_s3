# Цель 5: Паттерны проектирования (GoF)

## Теория (что спросят)

### Зачем паттерны?

Паттерны решают **типовые проблемы** проектирования. Связь с SOLID:

| Принцип SOLID | Какой паттерн помогает |
|---------------|----------------------|
| **S** — Single Responsibility | Command (разделяет действие и его выполнение) |
| **O** — Open/Closed | Strategy (новые алгоритмы без изменения кода) |
| **L** — Liskov Substitution | Composite (FileItem и FolderItem взаимозаменяемы) |
| **I** — Interface Segregation | Все паттерны через абстрактные интерфейсы |
| **D** — Dependency Inversion | Factory (зависимость от абстракции, не от конкретики) |

## 4 паттерна в проекте

### 1. Command — Undo/Redo действий

**Проблема**: пользователь удалил файл случайно. Как откатить?

**Решение**: каждое действие — это объект (Command) с методами `execute()` и `undo()`.

#### Структура

```
FileCommand (абстрактный)
├── DeleteFileCommand — удаление файла
├── MoveFileCommand — перемещение в папку
└── RenameFileCommand — переименование
```

**Файл**: `lib/features/files/domain/commands/file_command.dart`
```dart
abstract class FileCommand {
  String get description;
  Future<void> execute();
  Future<void> undo();
}
```

**Файл**: `lib/features/files/domain/commands/delete_file_command.dart`
```dart
class DeleteFileCommand implements FileCommand {
  final FilesLocalSource _source;
  final SecureFileEntity _file;
  SecureFileEntity? _backup;  // Сохраняем для undo

  @override
  Future<void> execute() async {
    _backup = _file;           // Запоминаем перед удалением
    await _source.delete(_file.id);
  }

  @override
  Future<void> undo() async {
    if (_backup != null) {
      await _source.save(_backup!);  // Восстанавливаем
    }
  }
}
```

**Файл**: `lib/features/files/domain/commands/command_manager.dart`
```dart
class CommandManager {
  final LinkedList<CommandEntry> _undoStack = LinkedList<CommandEntry>();
  final LinkedList<CommandEntry> _redoStack = LinkedList<CommandEntry>();
  static const int _maxHistorySize = 50;

  Future<void> execute(FileCommand command) async {
    await command.execute();
    _undoStack.add(CommandEntry(command));  // Добавляем в undo
    _redoStack.clear();                     // Redo сбрасывается

    if (_undoStack.length > _maxHistorySize) {
      _undoStack.first.unlink();            // Убираем самый старый
    }
  }

  Future<void> undo() async {
    final entry = _undoStack.last;
    entry.unlink();                          // Убираем из undo
    await entry.command.undo();
    _redoStack.add(CommandEntry(entry.command)); // В redo
  }

  Future<void> redo() async {
    final entry = _redoStack.last;
    entry.unlink();
    await entry.command.execute();
    _undoStack.add(CommandEntry(entry.command));
  }
}
```

**Как объяснить**: "Каждое деструктивное действие оборачивается в Command-объект. При execute() — действие выполняется и сохраняется в стек undo. При undo() — откатывается и переходит в стек redo. Стеки реализованы через LinkedList (Цель 6). Лимит — 50 команд."

**Как используется в BLoC**:
```dart
// FilesBloc
Future<void> _onDelete(FileDeleteRequested event, Emitter emit) async {
  final command = DeleteFileCommand(_localSource, event.file);
  await _commandManager.execute(command);
  add(FilesLoadRequested()); // Обновляем список
}

Future<void> _onUndo(UndoRequested event, Emitter emit) async {
  if (!_commandManager.canUndo) return;
  await _commandManager.undo();
  add(FilesLoadRequested());
}
```

### 2. Strategy — Сортировка файлов

**Проблема**: файлы можно сортировать по дате, имени, размеру, типу. Как менять алгоритм сортировки в runtime без if/else?

**Решение**: Strategy — интерфейс с разными реализациями. Алгоритм выбирается в runtime.

#### Структура

```
SortStrategy (абстрактный)
├── SortByDate — по дате создания (новые сверху)
├── SortByName — по имени (A-Z)
├── SortBySize — по размеру (большие сверху)
└── SortByType — по типу файла
```

**Файл**: `lib/features/files/domain/strategies/sort_strategy.dart`
```dart
abstract class SortStrategy {
  String get label;
  List<SecureFileEntity> sort(List<SecureFileEntity> files);
}
```

**Файл**: `lib/features/files/domain/strategies/sort_by_date.dart`
```dart
class SortByDate implements SortStrategy {
  @override
  String get label => 'By date';

  @override
  List<SecureFileEntity> sort(List<SecureFileEntity> files) =>
      [...files]..sort((a, b) => b.createdAt.compareTo(a.createdAt));
}
```

**Файл**: `lib/features/files/domain/strategies/sort_by_name.dart`
```dart
class SortByName implements SortStrategy {
  @override
  String get label => 'By name';

  @override
  List<SecureFileEntity> sort(List<SecureFileEntity> files) =>
      [...files]..sort((a, b) => a.name.toLowerCase().compareTo(b.name.toLowerCase()));
}
```

**Как используется в BLoC**:
```dart
SortStrategy _currentSortStrategy = SortByDate(); // По умолчанию

void _onSortStrategyChanged(SortStrategyChanged event, Emitter emit) {
  _currentSortStrategy = event.strategy; // Меняем стратегию в runtime
  final currentState = state;
  if (currentState is FilesLoaded) {
    final sorted = _currentSortStrategy.sort(currentState.files);
    emit(currentState.copyWith(files: sorted, sortStrategy: _currentSortStrategy));
  }
}
```

**Как объяснить**: "Вместо switch/case с логикой сортировки, у нас интерфейс SortStrategy. Каждая реализация знает как сортировать. BLoC хранит текущую стратегию и вызывает `.sort()`. Чтобы добавить новый вид сортировки — создаём новый класс, не трогая существующий код. Это Open/Closed Principle."

### 3. Factory — Превью файлов

**Проблема**: для разных типов файлов нужны разные превью-виджеты. Image → Image.memory, PDF → pdf viewer, Text → Text widget.

**Решение**: Factory создаёт нужный виджет по типу файла.

**Файл**: `lib/features/files/presentation/widgets/previews/file_preview_factory.dart`
```dart
abstract class FilePreviewFactory {
  static Widget create({
    required SecureFileEntity file,
    required Uint8List bytes,
  }) {
    switch (file.type) {
      case FileType.image:
        return ImagePreview(bytes: bytes);
      case FileType.pdf:
        return PdfPreview(bytes: bytes);
      case FileType.text:
        return TextPreview(bytes: bytes);
      case FileType.video:
      case FileType.audio:
      case FileType.unknown:
        return UnknownPreview(file: file);
    }
  }
}
```

**Как используется**:
```dart
// FileViewerScreen
final widget = FilePreviewFactory.create(file: file, bytes: decryptedBytes);
```

**Как объяснить**: "FilePreviewFactory скрывает логику создания превью. Вызывающий код не знает о конкретных виджетах (ImagePreview, PdfPreview) — только об абстракции. Чтобы поддержать новый тип файла — добавляем case в Factory и новый виджет."

### 4. Composite — Древовидная структура папок

**Проблема**: папка может содержать файлы и другие папки. Нужно рекурсивно считать размер и количество файлов.

**Решение**: Composite — одинаковый интерфейс для "листа" (файл) и "контейнера" (папка).

#### Структура

```
FileSystemItem (абстрактный)
├── FileItem (лист) — один файл, totalFiles = 1
└── FolderItem (контейнер) — содержит List<FileSystemItem>
    ├── FileItem
    ├── FileItem
    └── FolderItem (вложенная папка)
        └── FileItem
```

**Файл**: `lib/features/folders/domain/models/file_system_item.dart`
```dart
abstract class FileSystemItem {
  String get id;
  String get name;
  int get totalSize;
  int get totalFiles;
}
```

**Файл**: `lib/features/folders/domain/models/file_item.dart`
```dart
class FileItem implements FileSystemItem {
  final SecureFileEntity entity;

  @override
  int get totalSize => entity.size;

  @override
  int get totalFiles => 1;  // Файл — это всегда 1
}
```

**Файл**: `lib/features/folders/domain/models/folder_item.dart`
```dart
class FolderItem implements FileSystemItem {
  final List<FileSystemItem> children;

  @override
  int get totalSize =>
      children.fold<int>(0, (sum, item) => sum + item.totalSize);
      // Рекурсивно суммирует размеры всех вложенных элементов

  @override
  int get totalFiles =>
      children.fold<int>(0, (sum, item) => sum + item.totalFiles);
      // Рекурсивно считает все файлы
}
```

**Как объяснить**: "FileItem и FolderItem реализуют один интерфейс FileSystemItem. FolderItem содержит List<FileSystemItem> — может включать и файлы, и другие папки. Метод totalSize использует fold() для рекурсивного подсчёта: для файла это просто size, для папки — сумма totalSize всех children. Это Composite паттерн — клиентский код работает одинаково с файлами и папками."

## Сводка: когда какой паттерн

| Паттерн | Проблема | Решение | SOLID |
|---------|----------|---------|-------|
| **Command** | Undo/Redo действий | Действие = объект с execute()/undo() | SRP |
| **Strategy** | Разные алгоритмы сортировки | Интерфейс + реализации, переключение в runtime | OCP |
| **Factory** | Разные превью по типу файла | Централизованное создание виджетов | DIP |
| **Composite** | Дерево папок/файлов | Одинаковый интерфейс для листа и контейнера | LSP |

## Возможные вопросы на ассесменте

**В**: Зачем Command, если можно просто удалить и восстановить?
**О**: Command инкапсулирует действие. Добавить новое действие (переименование, перемещение) — создаём новый класс. Не трогаем CommandManager. Плюс — автоматически получаем Undo/Redo для любого нового действия.

**В**: Почему Strategy, а не просто switch/case в BLoC?
**О**: Switch/case нарушает Open/Closed — для нового вида сортировки нужно менять BLoC. С Strategy — создаёшь новый класс (SortBySize) и передаёшь в BLoC. Сам BLoC не меняется.

**В**: Чем Composite отличается от обычного дерева?
**О**: Composite — это паттерн, где "лист" (FileItem) и "узел" (FolderItem) реализуют одинаковый интерфейс. Клиентский код вызывает totalSize и не знает что внутри — файл или папка с 100 вложенными файлами.

**В**: Связь паттернов с SOLID?
**О**: Command — SRP (одна ответственность — одно действие). Strategy — OCP (открыт для расширения, закрыт для изменения). Factory — DIP (зависим от абстракции FilePreviewFactory, не от конкретных виджетов). Composite — LSP (FileItem и FolderItem взаимозаменяемы через FileSystemItem).
