# Цель 5: Продвинутые паттерны проектирования (GoF)

## Теория — зачем паттерны

Паттерны проектирования — **повторяемые решения** типичных архитектурных задач.
Не изобретаем велосипед, а используем проверенные подходы.

### Связь с SOLID

| Принцип | Паттерн | Как связаны |
|---------|---------|-------------|
| **S** (Single Responsibility) | Все 4 паттерна | Каждый класс — одна ответственность |
| **O** (Open-Closed) | Strategy, Factory | Добавить новое поведение без изменения существующего |
| **L** (Liskov Substitution) | Strategy, Composite | Подтипы заменяют базовый тип |
| **I** (Interface Segregation) | Command | Минимальный интерфейс: execute + undo |
| **D** (Dependency Inversion) | All | Зависим от абстракций, не реализаций |

## Паттерн 1: Strategy — Сортировка файлов

### Задача

Пользователь хочет сортировать файлы по дате, имени, размеру или типу.
Переключение в runtime через dropdown.

### UML-диаграмма

```
         ┌───────────────────┐
         │   SortStrategy     │  ← абстракция
         │   (abstract)       │
         ├───────────────────┤
         │ + label: String    │
         │ + sort(files): List│
         └────────┬──────────┘
                  │
    ┌─────────────┼─────────────┬─────────────┐
    │             │             │             │
┌───▼───┐  ┌─────▼─────┐  ┌───▼───┐  ┌─────▼─────┐
│SortBy │  │SortByName │  │SortBy │  │SortByType │
│Date   │  │           │  │Size   │  │           │
└───────┘  └───────────┘  └───────┘  └───────────┘
```

### Использование в BLoC

**Файл:** `lib/features/files/presentation/bloc/files_bloc.dart`

```dart
SortStrategy _currentSortStrategy = SortByDate();

void _onSortStrategyChanged(SortStrategyChanged event, Emitter emit) {
  _currentSortStrategy = event.strategy;

  final currentState = state;
  if (currentState is FilesLoaded) {
    final sorted = _currentSortStrategy.sort(currentState.files);
    emit(currentState.copyWith(files: sorted, sortStrategy: _currentSortStrategy));
  }
}
```

### Почему Strategy, а не switch/case

```dart
// ❌ Плохо — switch/case в BLoC
void _onSort(SortType type, Emitter emit) {
  switch (type) {
    case SortType.date: files.sort((a, b) => b.createdAt.compareTo(a.createdAt));
    case SortType.name: files.sort((a, b) => a.name.compareTo(b.name));
    case SortType.size: files.sort((a, b) => b.size.compareTo(a.size));
    // ... ещё 10 типов сортировки?
  }
}
```

Проблемы:
- Каждый новый тип сортировки → менять BLoC (**нарушает Open-Closed Principle**)
- BLoC раздувается сортировочной логикой
- Тестировать нужно весь BLoC ради одной сортировки

```dart
// ✅ С Strategy:
_currentSortStrategy = event.strategy; // подставляем ЛЮБУЮ реализацию
final sorted = _currentSortStrategy.sort(files); // полиморфизм
```

- Добавить `SortByTags` → создать новый класс, **не трогая BLoC**
- Тестировать каждую стратегию **изолированно**: `SortByDate().sort([file1, file2])`

### Расширяемость — пример

Если завтра нужна сортировка по тегам:

```dart
// Создаём новый класс — ВСЁ!
class SortByTags implements SortStrategy {
  @override
  String get label => 'By tags';

  @override
  List<SecureFileEntity> sort(List<SecureFileEntity> files) =>
      [...files]..sort((a, b) => a.tags.length.compareTo(b.tags.length));
}
```

BLoC, UI, другие стратегии — **ничего не меняем**.

## Паттерн 2: Command — Undo/Redo

### Задача

Пользователь случайно удалил файл → хочет отменить действие (Undo).
Отменил undo → хочет повторить (Redo).

### UML-диаграмма

```
┌──────────────────┐        ┌────────────────────┐
│  CommandManager   │        │  FileCommand        │ ← абстракция
│                   │        │  (abstract)         │
│ - undoStack       │ uses   ├────────────────────┤
│ - redoStack       │───────>│ + description       │
│ + execute(cmd)    │        │ + execute()         │
│ + undo()          │        │ + undo()            │
│ + redo()          │        └────────┬───────────┘
└───────────────────┘                 │
                           ┌──────────┼──────────┐
                           │          │          │
                    ┌──────▼──┐ ┌────▼─────┐ ┌──▼────────┐
                    │DeleteFile│ │MoveFile  │ │RenameFile │
                    │Command   │ │Command   │ │Command    │
                    └──────────┘ └──────────┘ └───────────┘
```

### Реализация

**Файл:** `lib/features/files/domain/commands/file_command.dart`

```dart
abstract class FileCommand {
  String get description;
  Future<void> execute();
  Future<void> undo();
}
```

**Файл:** `lib/features/files/domain/commands/delete_file_command.dart`

```dart
class DeleteFileCommand implements FileCommand {
  final FilesLocalSource _source;
  final SecureFileEntity _file;
  SecureFileEntity? _backup;

  DeleteFileCommand(this._source, this._file);

  @override
  String get description => 'Delete "${_file.name}"';

  @override
  Future<void> execute() async {
    _backup = _file;              // сохраняем backup для undo
    await _source.delete(_file.id);
  }

  @override
  Future<void> undo() async {
    if (_backup != null) {
      await _source.save(_backup!); // восстанавливаем из backup
    }
  }
}
```

**Файл:** `lib/features/files/domain/commands/command_manager.dart`

```dart
@lazySingleton
class CommandManager {
  final LinkedList<CommandEntry> _undoStack = LinkedList<CommandEntry>();
  final LinkedList<CommandEntry> _redoStack = LinkedList<CommandEntry>();
  static const int _maxHistorySize = 50;

  bool get canUndo => _undoStack.isNotEmpty;
  bool get canRedo => _redoStack.isNotEmpty;

  Future<void> execute(FileCommand command) async {
    await command.execute();
    _undoStack.add(CommandEntry(command));
    _redoStack.clear();           // новое действие → redo недоступен

    if (_undoStack.length > _maxHistorySize) {
      _undoStack.first.unlink();  // O(1) — удаляем самую старую команду
    }
  }

  Future<void> undo() async {
    if (!canUndo) return;
    final entry = _undoStack.last;
    entry.unlink();               // убираем из undo стека
    await entry.command.undo();
    _redoStack.add(CommandEntry(entry.command)); // перемещаем в redo
  }

  Future<void> redo() async {
    if (!canRedo) return;
    final entry = _redoStack.last;
    entry.unlink();
    await entry.command.execute();
    _undoStack.add(CommandEntry(entry.command)); // обратно в undo
  }
}
```

### Поток Undo/Redo

```
Начало:  undoStack: []  redoStack: []

1. Удалить файл A:
   execute(DeleteA)
   undoStack: [DeleteA]  redoStack: []

2. Удалить файл B:
   execute(DeleteB)
   undoStack: [DeleteA, DeleteB]  redoStack: []

3. Undo (отменяем удаление B):
   undo()  → B.undo() → restore B
   undoStack: [DeleteA]  redoStack: [DeleteB]

4. Redo (повторяем удаление B):
   redo()  → B.execute() → delete B again
   undoStack: [DeleteA, DeleteB]  redoStack: []

5. Undo + новое действие:
   undo()  → B.undo()
   undoStack: [DeleteA]  redoStack: [DeleteB]
   execute(DeleteC)  → redoStack.clear()!
   undoStack: [DeleteA, DeleteC]  redoStack: []
```

### Почему Command

- **Инкапсуляция:** команда = действие + отмена + описание в одном объекте
- **Расширяемость:** `MoveFileCommand`, `RenameFileCommand` — каждый знает свой undo
- **Единый менеджер:** `CommandManager` не знает деталей команд — работает через `execute()/undo()`
- **Testability:** `DeleteFileCommand` тестируется отдельно от BLoC

## Паттерн 3: Factory — Превью файлов

### Задача

При открытии файла нужно показать подходящий виджет:
- Изображение → `ImagePreview`
- PDF → `PdfPreview`
- Текст → `TextPreview`
- Прочее → `UnknownPreview`

### Реализация

**Файл:** `lib/features/files/presentation/widgets/previews/file_preview_factory.dart`

```dart
abstract class FilePreviewFactory {
  static Widget create({
    required SecureFileEntity file,
    required Uint8List bytes,
  }) {
    return switch (file.type) {
      FileType.image => ImagePreview(bytes: bytes),
      FileType.pdf => PdfPreview(bytes: bytes),
      FileType.text => TextPreview(bytes: bytes),
      FileType.video || FileType.audio || FileType.unknown =>
          UnknownPreview(file: file),
    };
  }
}
```

### Почему Factory

`FileViewerScreen` **не знает**, какой виджет нужен. Делегирует выбор фабрике:

```dart
// В FileViewerScreen:
final preview = FilePreviewFactory.create(file: file, bytes: decryptedBytes);
```

Вызывающий код одинаков для любого типа файла. Логика выбора — в одном месте.

### Альтернатива — if/else в build()

```dart
// ❌ UI загрязнён логикой выбора
Widget build(context) {
  if (file.type == FileType.image) return ImagePreview(...);
  else if (file.type == FileType.pdf) return PdfPreview(...);
  // ... повторять в каждом месте где нужен preview
}
```

С Factory — логика выбора **централизована** и **переиспользуема**.

### Dart 3 pattern matching

Используем `switch expression` (Dart 3) вместо `switch statement`:

```dart
// Dart 3 switch expression — exhaustive (компилятор проверяет все варианты)
return switch (file.type) {
  FileType.image => ImagePreview(bytes: bytes),
  FileType.pdf => PdfPreview(bytes: bytes),
  FileType.text => TextPreview(bytes: bytes),
  FileType.video || FileType.audio || FileType.unknown => UnknownPreview(file: file),
};
```

Если добавить `FileType.archive` в enum — **компилятор** выдаст ошибку:
"The type 'FileType' is not exhaustively matched". Не забудем добавить обработку.

## Паттерн 4: Composite — Древовидная структура папок

### Задача

Файлы и папки образуют дерево. Папка может содержать и файлы, и подпапки.
Нужен общий интерфейс для подсчёта общего размера и количества файлов.

### UML-диаграмма

```
         ┌──────────────────┐
         │  FileSystemItem   │ ← абстракция (компонент)
         │  (abstract)       │
         ├──────────────────┤
         │ + id: String      │
         │ + name: String    │
         │ + totalSize: int  │
         │ + totalFiles: int │
         └────────┬─────────┘
                  │
         ┌────────┴────────┐
         │                 │
    ┌────▼─────┐    ┌─────▼──────┐
    │ FileItem  │    │ FolderItem  │ ← composite
    │ (leaf)    │    │             │
    ├──────────┤    ├─────────────┤
    │ totalSize │    │ children:   │
    │ = entity  │    │  List<FSItem>│
    │   .size   │    │             │
    │ totalFiles│    │ totalSize = │
    │ = 1       │    │  children   │
    └──────────┘    │  .fold(...)  │
                     │ totalFiles =│
                     │  children   │
                     │  .fold(...)  │
                     └─────────────┘
```

### Реализация

**Файл:** `lib/features/folders/domain/models/file_system_item.dart`

```dart
abstract class FileSystemItem {
  String get id;
  String get name;
  int get totalSize;
  int get totalFiles;
}
```

**Файл:** `lib/features/folders/domain/models/file_item.dart`

```dart
class FileItem implements FileSystemItem {

  const FileItem(this.entity);

  @override
  String get id => entity.id;
  @override
  String get name => entity.name;
  @override
  int get totalSize => entity.size;
  @override
  int get totalFiles => 1;  // файл = 1 файл
}
```

**Файл:** `lib/features/folders/domain/models/folder_item.dart`

```dart
class FolderItem implements FileSystemItem {
  final List<FileSystemItem> children;
  // ...

  @override
  int get totalSize =>
      children.fold<int>(0, (sum, item) => sum + item.totalSize);

  @override
  int get totalFiles =>
      children.fold<int>(0, (sum, item) => sum + item.totalFiles);
}
```

### Рекурсия через fold

```
Folder "Documents" (children: [
  FileItem("photo.jpg", 5MB),
  FolderItem("Work", children: [
    FileItem("report.pdf", 3MB),
    FileItem("notes.txt", 1MB),
    FolderItem("Archive", children: [
      FileItem("old.zip", 10MB),
    ]),
  ]),
])

totalSize:
  Documents.totalSize
    = 5 + Work.totalSize
    = 5 + (3 + 1 + Archive.totalSize)
    = 5 + (3 + 1 + 10)
    = 19 MB

totalFiles:
  Documents.totalFiles
    = 1 + Work.totalFiles
    = 1 + (1 + 1 + Archive.totalFiles)
    = 1 + (1 + 1 + 1)
    = 4 файла
```

### Почему Composite

- **Единый интерфейс:** `FileItem` и `FolderItem` — оба `FileSystemItem`
- **Рекурсия прозрачна:** `folder.totalSize` автоматически считает вложенные
- **Полиморфизм:** UI работает с `FileSystemItem`, не знает file это или folder
- **Liskov Substitution:** подставляем `FileItem` или `FolderItem` — всё работает

### Альтернатива — отдельные классы без общего интерфейса

```dart
// ❌ Без Composite:
int calculateSize(dynamic item) {
  if (item is File) return item.size;
  if (item is Folder) {
    var total = 0;
    for (final child in item.children) {
      total += calculateSize(child); // dynamic → нет type safety
    }
    return total;
  }
  return 0;
}
```

С Composite: `item.totalSize` — type-safe, элегантно, рекурсия в самом классе.

## Сводка паттернов

| Паттерн | Тип | Где используется | Какую проблему решает |
|---------|-----|------------------|-----------------------|
| Strategy | Behavioral | Сортировка файлов | Переключение алгоритма в runtime |
| Command | Behavioral | Undo/Redo удаления | Инкапсуляция действия + отмена |
| Factory | Creational | Превью по типу файла | Централизованное создание объектов |
| Composite | Structural | Дерево папок | Единый интерфейс для leaf/composite |
